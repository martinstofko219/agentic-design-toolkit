This skill generates three genuinely distinct wireframe concepts for a given design problem, rendered as an interactive HTML file. Each variation shows a 2–4 screen flow with clickable navigation. The goal is rapid concept exploration — breadth over polish, structure over style. Each variation should represent a meaningfully different bet on how to structure the experience: a different information architecture, a different interaction model, or a fundamentally different mental model for the user.

## Before You Start

If the designer's brief is thin — no clear problem statement, no defined user, or genuine uncertainty about what they're trying to solve — suggest running the `design-grill` skill first. Wireframes are only as good as the thinking behind them, and a short interrogation now saves a lot of wasted exploration later.

Ask clarifying questions when any of the following are unknown. Don't generate wireframes without this foundation.

Always ask:
- **Platform / screen type**: Mobile, desktop, tablet, or multi-platform?
- **The user and their goal**: Who is this for, and what are they trying to accomplish?

Ask if not provided or clearly inferable:
- **Problem context**: What design problem is being solved?
- **Key constraints**: Technical, accessibility, brand, or content constraints worth respecting.
- **Content mode**: Should body copy use placeholder lines (default) or realistic text written for the industry?

Skip questions the designer has already answered.

## Generating Three Distinct Variations

The three variations must represent genuinely different design bets — not the same layout reshuffled. Aim for difference at the level of information architecture, interaction model, or mental model; a useful test is whether one-sentence descriptions of each would sound like genuinely different approaches. Label each with a name that captures its bet — "Progressive Disclosure", "Single Canvas", "Guided Wizard" — not "Option 1." Each variation includes 2–4 connected screens chosen to tell the full story of the interaction model being explored.

## Visual Style

Aim for a warm, hand-crafted lo-fi aesthetic — something between a well-made paper prototype and a thoughtful Figma wireframe. It should feel organic and intentional, not sterile. Use warm off-white and cream tones for backgrounds and surfaces, near-black warm ink for text and borders, and one accent color to signal interactivity or alert states. Typography should lean handwritten — think sketch fonts for headings and a looser body font for content, with a monospaced option for metadata and codes. Borders on containers should feel slightly imprecise, like they were drawn by hand rather than generated by a grid. Image and chart placeholders should use a subtle crosshatch or hatch fill, not just a flat gray box. The overall impression should be: "this is a thoughtful sketch, not a finished product." Avoid pure grays and clinical whites — warmth is the point.

In *placeholder mode* (default), represent body copy as short horizontal bars of varying widths suggesting lines of text. Use real words only for headings, labels, CTAs, and navigation.

In *realistic copy mode*, write genuine content appropriate to the problem's industry. The copy should feel like it came from the actual product.

## Output

Deliver a single self-contained `.html` file that opens in any browser with no build step. Structure it in layers: styles and fonts, shared primitive components (Box, ImagePlaceholder, Chip, TextLines), domain-specific components for the problem, variation screen functions, and finally the app shell.

The app shell must include:

**Brief panel** (visible by default): shows the problem statement, platform, constraints, and for each variation: its name, design bet in 1–2 sentences, and a simple screen flow map (e.g. "Dashboard → Detail → Edit").

**Variation tabs**: one tab per concept. Clicking switches the flow player to that variation.

**Flow player**: renders the current screen at full fidelity with a screen label ("Screen 2 of 3 — Medication Detail") and Previous / Next navigation. Clickable elements within the wireframe can also advance to the next screen.

**Annotation panel**: toggleable right-side panel (roughly 65/35 split with the wireframe). Shows 3–5 numbered annotations for the current screen, updating when the screen changes. Annotations explain design intent and the assumption being tested — not just labels.

## JavaScript Safety

All string values in JS data objects (screen content, annotation text, labels) must use double quotes or template literals — never single quotes, which break silently when the text contains an apostrophe (`"What's next"` is safe; `'What's next'` is a syntax error). This is the most common cause of blank screens.
